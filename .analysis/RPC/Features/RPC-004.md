# RPC-004: Switcher Driver Framework & Hardware Abstraction

## Kontekst biznesowy

ripcd musi komunikowac sie z 42+ roznymi modelami urzadzen audio (switchers, routery, karty GPIO) od roznych producentow (BroadcastTools, Logitek, Axia, Wheatstone, SAS, Ross, etc.). Kazdy urzadzenie ma wlasny protokol komunikacji (serial RS-232, TCP/IP, HTTP, multicast, Modbus, kernel GPIO). Switcher Driver Framework zapewnia abstrakcje -- wszystkie drivery dziedzicza z klasy bazowej Switcher i implementuja ten sam interfejs (processCommand, sendGpi/sendGpo, sygnaly GPIO). Factory pattern w LoadSwitchDriver dynamicznie tworzy odpowiedni driver na podstawie konfiguracji DB.

## Aktorzy

| Aktor | Rola w tej feature |
|-------|-------------------|
| Administrator | Konfiguruje macierze w DB (typ, adres, port) |
| Sprzet audio | Urzadzenia komunikujace sie z driverami |
| ripcd (MainObject) | Tworzy drivery, deleguje komendy |

## Granica funkcjonalnosci

```
IN SCOPE:
  - Switcher abstract base class (interfejs + wspolna logika)
  - LoadSwitchDriver() factory method
  - 42 driver subclasses (implementacje per producent/model)
  - StarGuideFeed, UnityFeed (helper DTOs)
  - Polaczenia sygnalow driver <-> MainObject (loaddrivers.cpp)
  - Wzorzec serial driver: RDTTYDevice, QSignalMapper, readyReadData
  - Wzorzec TCP driver: QTcpSocket, connected/disconnected/readyRead/error, watchdog
  - Wzorzec local driver: poll timer, HPI/JACK/Kernel GPIO
  - RML SZ (reload driver)
  - processCommand dispatch (CL, FS, GO, ST, SA, SD, SG, SR, SL, SX)
OUT OF SCOPE:
  - GPIO state management -> patrz RPC-002
  - TTY port management -> patrz RPC-005 (drivery uzywaja TTY ale nim nie zarzadzaja)
  - JACK port management -> patrz RPC-006
```

## Reguly biznesowe (Gherkin)

```gherkin
Rule: Factory pattern

  Scenario: Ladowanie drivera na starcie
    Given numer macierzy matrix_num, konfiguracja w MATRICES
    When  LoadSwitchDriver(matrix_num) wywolane
    Then  tworzy RDMatrix z DB
    And   switch(matrix->type()) tworzy odpowiedni Switcher subclass
    And   connect 5 sygnalow: rmlEcho, gpiChanged, gpoChanged, gpiState, gpoState
  # Zrodlo: loaddrivers.cpp:72-279 | Pewnosc: potwierdzone

  Scenario: Nieznany typ drivera
    Given RDMatrix::Type nie pasuje do zadnego case
    When  LoadSwitchDriver() wywolane
    Then  zwraca false, switcher[matrix] pozostaje NULL
  # Zrodlo: loaddrivers.cpp | Pewnosc: potwierdzone

Rule: Delegacja komend switcher

  Scenario: Komenda do zaladowanego drivera
    Given macierz matrix z zaladowanym driverem
    When  RML CL/FS/GO/ST/SA/SD/SG/SR/SL/SX odebrane
    Then  ripcd_switcher[matrix]->processCommand(rml)
  # Zrodlo: local_macros.cpp:814-840 | Pewnosc: potwierdzone

  Scenario: Komenda do niezaladowanej macierzy
    Given ripcd_switcher[matrix] == NULL
    When  komenda switcher odebrane
    Then  acknowledge(false) + sendRml
  # Zrodlo: local_macros.cpp:831-836 | Pewnosc: potwierdzone

Rule: Hot-reload drivera (RML SZ)

  Scenario: Przeladowanie drivera
    Given macierz matrix z istniejacym driverem
    When  RML "SZ matrix" odebrane
    Then  zwolnij TTY porty powiazane z macierza
    And   delete stary driver
    And   LoadSwitchDriver(matrix) z nowa konfiguracja
  # Zrodlo: local_macros.cpp:903-932 | Pewnosc: potwierdzone

Rule: Wspierany interfejs Switcher

  Scenario: Kazdy driver implementuje 6 pure virtual methods
    Given nowa implementacja drivera
    When  driver dziedziczy z Switcher
    Then  MUSI implementowac: type(), gpiQuantity(), gpoQuantity(),
          primaryTtyActive(), secondaryTtyActive(), processCommand()
  # Zrodlo: switcher.h:38-46 | Pewnosc: potwierdzone
```

## API klas w scope

### Switcher (abstract base)

**Odpowiedzialnosc:** Abstrakcyjna baza -- unifikuje interfejs 42 driverow.

**Pure virtual methods:**

| Metoda | Zwraca | Opis |
|--------|--------|------|
| type() | RDMatrix::Type | Identyfikator typu drivera |
| gpiQuantity() | unsigned | Liczba pinow GPI |
| gpoQuantity() | unsigned | Liczba pinow GPO |
| primaryTtyActive() | bool | Czy glowny TTY aktywny |
| secondaryTtyActive() | bool | Czy dodatkowy TTY aktywny |
| processCommand(cmd) | void | Obsluz komende RML |

**Virtual methods (default):**

| Metoda | Opis |
|--------|------|
| sendGpi() | Emit gpiState dla kazdego pinu (default: no-op) |
| sendGpo() | Emit gpoState dla kazdego pinu (default: no-op) |

**Sygnaly:**

| Sygnal | Parametry | Znaczenie |
|--------|-----------|-----------|
| rmlEcho(RDMacro*) | cmd | Driver chce odpowiedziec na komende |
| gpiChanged(int,int,bool) | matrix, line, state | GPI zmienil stan |
| gpoChanged(int,int,bool) | matrix, line, state | GPO zmienil stan |
| gpiState(int,unsigned,bool) | matrix, line, state | Pelny raport GPI |
| gpoState(int,unsigned,bool) | matrix, line, state | Pelny raport GPO |

**Protected methods:**

| Metoda | Opis |
|--------|------|
| executeMacroCart(cartnum) | Uruchom makro cart via emit rmlEcho |
| logBytes(data, len) | Debug log surowych bajtow |
| insertGpioEntry(is_gpo, line) | INSERT do GPIO_EVENTS |
