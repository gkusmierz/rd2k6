# LIB-002: Log & Playlist Data Model

## Kontekst biznesowy

Log (playlist) jest centralnym artefaktem emisji radiowej. Definiuje co, kiedy i w jakiej kolejnosci jest emitowane. Sklada sie z linii (LogLine) ktore moga byc cartami audio, makrami, markerami, linkami do importow music/traffic. Model danych logu musi obslugiwac operacje edycyjne (insert, remove, move, copy) oraz walidacje.

## Aktorzy

| Aktor | Rola |
|-------|------|
| Operator | Edytuje log, wstawia/usuwa elementy |
| System | Laduje, zapisuje, waliduje log |

## Granica funkcjonalnosci

```
IN SCOPE:
  - Encja Log: CRUD, metadane, link state
  - Encja LogEvent: kontener linii, operacje edycyjne
  - Encja LogLine: 9 typow linii, przejscia, markery, metadane
  - Encja LogLock: blokada pesymistyczna
  - Walidacja logu

OUT OF SCOPE:
  - Odtwarzanie logu (LIB-003)
  - Generowanie logu z zegarow (LIB-006)
```

## Reguly biznesowe (Gherkin)

```gherkin
Rule: Log obsluguje 9 typow linii
  Scenario: Linia typu Cart
    Given linia z type=Cart i cartNumber=100001
    When ladowana jest z DB
    Then metadane carta sa dolaczone

Rule: Przejscia miedzy elementami
  Scenario: TransType definiuje jak elementy sa laczone
    Given linia z transType=Segue
    When nastepna linia jest odtwarzana
    Then obie linie nakladaja sie czasowo

Rule: Log lock zapobiega jednoczesnemu edytowaniu
  Scenario: Lock timeout
    Given log jest zablokowany przez uzytkownika A
    When uzytkownik B probuje edytowac po 30s
    Then lock wygasa i B moze edytowac
```

## Dane i encje domenowe

| Encja | Operacje | Pola wymagane |
|-------|----------|--------------|
| Log | CRUD, updateTracks, removeTracks | name, service |
| LogEvent | load, save, insert, remove, move, copy, validate | logName |
| LogLine | CRUD, loadCart, resolveWildcards | id, type |
| LogLock | lock, unlock, refresh | logName |
