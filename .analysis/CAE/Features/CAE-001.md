# CAE-001: TCP Server & Protocol Layer

## Kontekst biznesowy

CAE musi przyjmowac polaczenia od wielu klientow jednoczesnie i obslugiwac tekstowy protokol komend. Warstwa serwerowa TCP jest fundamentem calego daemona -- bez niej zadne sterowanie audio nie jest mozliwe. Obejmuje nasluchiwanie, autoryzacje haslem, parsowanie komend i routing do odpowiednich slotow.

## Aktorzy

| Aktor | Rola w tej feature |
|-------|-------------------|
| Klient TCP (rdairplay, rdlibrary, etc.) | Laczy sie, autoryzuje, wysyla komendy |
| System (daemon) | Nasluchuje na porcie, zarzadza polaczeniami |

## Granica funkcjonalnosci

```
IN SCOPE:
  - Nasluchiwanie TCP na CAED_TCP_PORT
  - Przyjmowanie nowych polaczen (multi-client)
  - Autoryzacja haslem (komenda PW)
  - Parsowanie protokolu tekstowego (komendy terminowane '!')
  - Routing komend do sygnalow Qt
  - Wysylanie odpowiedzi (unicast i broadcast)
  - Obsluga rozlaczenia (DC, socket closed)
  - Cleanup zasobow przy rozlaczeniu klienta

OUT OF SCOPE:
  - Implementacja operacji audio → patrz CAE-002, CAE-003, CAE-004
  - Metering UDP → patrz CAE-005
  - Startup i provisioning → patrz CAE-006
```

## Reguly biznesowe (Gherkin)

```gherkin
Rule: Autoryzacja klienta haslem

  Scenario: Klient podaje poprawne haslo
    Given klient polaczony przez TCP
    When  klient wysyla "PW <password>!"
    And   password == rd_config->password()
    Then  klient jest autoryzowany (authenticated=true)
    And   odpowiedz "PW +!"

  Scenario: Klient podaje bledne haslo
    Given klient polaczony przez TCP
    When  klient wysyla "PW <bledne_haslo>!"
    Then  klient NIE jest autoryzowany
    And   odpowiedz "PW -!"

  Scenario: Klient wysyla komende bez autoryzacji
    Given klient nie autoryzowany
    When  klient wysyla komende inna niz PW lub DC
    Then  komenda jest ignorowana
  # Zrodlo: cae_server.cpp:226-244 | Pewnosc: potwierdzone

Rule: Parsowanie protokolu

  Scenario: Poprawna komenda
    Given klient autoryzowany
    When  klient wysyla "LP 0 myfile!"
    Then  komenda podzielona na tokeny po spacjach
    And   parametry zwalidowane (card < RD_MAX_CARDS, etc.)
    And   odpowiedni sygnal emitowany (loadPlaybackReq)

  Scenario: Bledna komenda
    Given klient autoryzowany
    When  klient wysyla nierozpoznana komende
    Then  odpowiedz "{komenda}-!" (generic error)
  # Zrodlo: cae_server.cpp:202-563 | Pewnosc: potwierdzone

Rule: Cleanup przy rozlaczeniu

  Scenario: Klient z aktywnymi streamami rozlacza sie
    Given klient ma aktywne playbacki/nagrywania
    When  socket zamkniety lub DC
    Then  KillSocket() iteruje ALL karty x streamy
    And   streamy tego klienta sa zwalniane
    And   ownery resetowane do -1
  # Zrodlo: cae.cpp:1739-1800 | Pewnosc: potwierdzone
```

## Protokoly komunikacji

Pelna referencja protokolu CAE: patrz `SPEC.md Sekcja 9`

Kluczowe komendy dla tej feature:

| Komenda | Parametry | Odpowiedz | Znaczenie |
|---------|-----------|-----------|-----------|
| PW | password | PW +! / PW -! | Autoryzacja |
| DC | - | (disconnect) | Rozlaczenie |

Format protokolu:
- Komendy: 2-literowy token + parametry + "!" (terminator)
- Odpowiedzi sukcesu: "{CMD} {params} +!"
- Odpowiedzi bledu: "{CMD} {params} -!"
- Separator: '!' (wykrzyknik), nie newline
- Parsowanie: split po spacjach, walidacja kazdego parametru
