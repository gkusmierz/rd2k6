# LIB-007: IPC & Communication

## Kontekst biznesowy

Komunikacja miedzyprocesowa jest kregoslupem systemu. RDCae steruje serwerem audio (caed), RDRipc komunikuje sie z serwerem RPC (ripcd) — GPIO, powiadomienia, RML. RDCatchConnect laczy sie z schedulerem nagrywania (rdcatchd). System powiadomien (RDNotification) synchronizuje stan miedzy wszystkimi aplikacjami.

## Aktorzy

| Aktor | Rola |
|-------|------|
| System | Komunikacja miedzy procesami, synchronizacja |
| Operator | Posredni — akcje operatora generuja komendy |

## Granica funkcjonalnosci

```
IN SCOPE:
  - RDCae: klient audio engine (play/stop/record/meters)
  - RDRipc: klient RPC (GPIO, RML, notifications, on-air)
  - RDCatchConnect: klient catch (zaplanowane zadania)
  - RDNotification: powiadomienia broadcast
  - RDMacro + RDMacroEvent: komendy RML
  - RDSocket, RDUnixServer, RDUnixSocket: transport
  - RDMulticaster: multicast UDP

OUT OF SCOPE:
  - Implementacja demonow (artefakty CAE, RPC, CTD)
  - Playback engine (LIB-003)
```

## Reguly biznesowe (Gherkin)

```gherkin
Rule: Powiadomienia sa broadcast do wszystkich klientow
  Scenario: Modyfikacja carta
    Given 3 aplikacje polaczone z ripcd
    When app1 wysyla Notification(CartType, Modify, 100001)
    Then app2 i app3 otrzymuja to powiadomienie

Rule: Komendy RML (60 komend dwuliterowych)
  Scenario: Komenda Play
    Given komenda PL z argumentami (machine, line, trans)
    When dostarczona do aplikacji
    Then wskazana linia logu jest odtwarzana

Rule: Heartbeat monitoruje polaczenie
  Scenario: Utrata heartbeat
    Given polaczenie z rdcatchd
    When heartbeat nie dotrze w 30s
    Then emit heartbeatFailed()
```
